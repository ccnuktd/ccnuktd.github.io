Log4Shell的反思

Log4Shell漏洞反思

作为一个计算机人，我经常会浏览一些科技热点，前段时间电脑就被一个号称 【核弹级漏洞】 的log4Shell漏洞刷屏了。那么究竟Log4Shell漏洞是什么呢？

我参考了网上少数派的 一篇报道 ，写了一些自己的理解。虽然在此之前，我完全不懂信息安全类的知识，但是本着处处都可以学习的原则，写了一篇博客用来记录。

Log4Shell漏洞是什么？

log4Shell 漏洞涉及到 log4j 工具。Log4j是一个用来记录日志的 Java 工具，它的工作就是将访问服务器的信息记录下来，但是这里的记录并不止是最简单的记录，还涉及到查询的功能，这也就是漏洞的运作的原理了。

Log4j虽然只是一个日志记录的工具，但是它的权限可不少，它具有查询的功能。之所以会有这个功能是设计者考虑到日志记录可能会发生这种情况：

正常情况下，如果有一次访问，访问者需要提供自己具体的信息，log4j 记录下来就好了。

但是还有一种情况是访问者不提供自己的全部信息，这时 log4j 就会通过一些检索、计算、查询来补全信息。

这里就需要引入 LDAP和 JNDI 。

LDAP （轻型目录访问协议，Lightweight Directory Access Protocol）是一种记录身份信息的协议。 log4j 记录信息就需要遵循这个协议。而协议支持 JNDI 查询方式，JNDI（Java Naming and Directory Interface）可以在给定的一个目录中查找信息，如果提供类似于env.put(Context.PROVIDER_URL, "ldap://127.0.0.1:389"); 这样的IP地址，就可以去查询里面的具体信息。

那么如果在 log4j 记录日志时，提供一个恶意的IP地址， log4j 在补全信息的时候访问了这个恶意IP。看起来好像还是没什么问题，但是访问一个IP可不像我们平时点开一个网页那么随意，如果这个IP是一个病毒的下载地址，那么使用 log4j 的服务器就会下载一个病毒，这样问题就很大了。除此之外，访问IP地址时，也需要提供自己的IP地址，那么服务器的内部IP就会泄露出去，甚至许多别的重要的信息也会泄露出去，这样就让黑客有机可乘了。

漏洞背后的思考

在 报道 中提到， log4j 的权限是不是太大了，区区一个记录员却可以访问未经鉴别的外部地址。正常情况下记录不全，应该交给专门的组件来补全信息，而不是自己擅作主张。作为软件的开发者，这个锅得由他背。

但是我们反过来思考一下。 log4j 是一个开源软件。既然是开源的，由成千上万企业使用的软件，为什么会出现如此低级的问题，而且经过了这么久才被人发现。Log4j 此次漏洞也让人反思开源维护者是否得到了应有的支持和理解。

借用 报道 中的论点，既然 Log4j 的使用如此广泛、在各大主流服务中任劳任怨，那么大厂的担当和风范何在？因此有观点 主张 ，使用开源项目的公司有道德上的责任赞助和支持项目的维护者；还有人 提出 ，大厂即使不提供金钱支持，是不是至少应该义务提供技术力量，辅助改进整个项目，而不是自扫门前雪，修好自己的服务了事？

那么就可以过渡到今天的第二个话题：

如何看待faker.js开源作者删库跑路？

参考 机器之心的报道，1月5日Marak将自己开发的faker.js删除。

并向自己开发的colors.js提交恶意代码，导致其无法使用。

提交代码后，如果使用该库会出现各种乱码。

目前，color.js 已经更新了一个可以使用的版本。faker.js 项目尚未恢复，开发者只能通过降级到此前的 5.5.3 版本来解决问题。

引用 developer网的报道，Marak在之前的一个月，他所居住的的公寓发生火灾，他失去了全部家当。

再加上Marak作为一个杰出的开发者，无偿开源自己的代码，但是自己却连一个可观的offer都拿不到，大公司白嫖他的代码，却不给他养活自己的机会，这着实令人唏嘘。（图片引用）

本来做开源是一件无私奉献的事情，但是这样的境遇未免会使作者寒心。就和log4j工具一样，提供工具的开发者不仅没有收到应得的东西，而且受到使用工具的人们的大量指责。这不禁让人怀疑还会有人再去做开源项目吗？

虽然这两件发生的事情没有直接的关联，但是背后反应的确是同一个问题。如何让提供开源项目的作者不既流汗又受罪，这是我们每一个使用者需要思考的问题。
